15.4 工程师的每日工作流

上午：加载上下文

在写代码之前，先为 AI 加载上下文：

# 在 Cursor/Claude Code 中可用的模式
@workspace  # Cursor：加载完整仓库上下文
@context src/api/ src/models/ tests/  # 指定目录

推荐实践： 在仓库根目录维护一个 AGENTS.md（或 .cursorrules、CLAUDE.md）文件：

# AGENTS.md — AI 编程助手的项目指南

## 技术栈
- 前端：Next.js 15、TypeScript、Tailwind、shadcn/ui
- 后端：FastAPI、Python 3.12、SQLAlchemy 2.0
- 数据库：PostgreSQL 16、Redis 缓存
- 测试：pytest、Playwright、Vitest

## 约定
- 使用函数式组件 + hooks，不使用类组件
- 所有 API 路由都有 Zod 验证
- 新业务逻辑必须包含测试
- 错误信息格式："ERR_XXXX: description"

## 架构
- MVC + service 层
- Repositories 抽象数据库访问
- 全程使用 async/await

中午：生成

最大化效率的编码工作流：

  1. 规格先行 — 在写函数体之前先写接口/签名/类型。当契约清晰时，AI 生成的代码质量更高。

  2. 测试先行 — 先写测试。然后让 AI 实现。这锚定了正确性。

  3. 迭代精炼 — 不要接受第一个输出。利用 AI 的自我精炼能力：

    "审查这段代码，关注：边界情况、性能问题、安全漏洞和风格一致性。"

  4. 上下文锚定 — 直接将编译器/解释器错误粘贴到提示中。AI 在看到错误时修复自身输出的能力特别强。

有效的提示模式：

模式示例为何有效
角色分配“你是一名正在审查代码库的高级 Python 工程师”激活专家角色
约束前缀“写生产级代码。这段代码将处理 PII 数据。”设定安全边界
输出格式“以相对于 main 分支的 diff 形式返回解决方案”减少认知负担
负面示例“不要使用 eval、subprocess 或 exec”预防常见失败模式
思维链“在写代码之前，先思考边界情况”提高正确性
自我验证“编写测试验证这段代码，然后运行它们”创建检查点

下午：审查与重构

AI 辅助代码审查清单：

  1. 逻辑正确性 — “是否处理了 X 为 null 的边界情况？”
  2. 安全扫描 — “检查 OWASP Top 10 漏洞：注入、XSS、SSRF、不安全反序列化”
  3. 性能 — “找出 O(n²) 或更差的模式、N+1 查询、内存泄漏”
  4. 可维护性 — “这个函数太长了吗？有没有魔法数字？命名一致吗？”
  5. 测试充分性 — “哪些场景没有被现有测试覆盖？”

重构模式：

"分析这个模块。找出：
1. 应该提取的重复逻辑
2. 违反单一职责原则的函数
3. 可以降低的耦合度
4. 使用现代语言特性的机会
输出一个按优先级排序的重构计划。"

15.5 AI 用于测试

测试是 AI 提供最高 ROI 的领域之一。

测试生成

工具方法质量
Qodo（CodiumAI）静态分析 → 行为覆盖边界检测优秀
Copilot / Cursor行内测试生成单元测试表现良好
Coverage AILLM 分析未覆盖分支生成缺失场景
Property-based（Hypothesis + AI）AI 建议不变量发现边界 Bug

推荐工作流：

# 第一步：从实现生成测试
# 提示："为这个函数生成 pytest 测试，覆盖：
# - 正常路径
# - 边界情况（空输入、None、边界值）
# - 错误条件
# - 类型违规"

# 第二步：审查并运行
# - AI 经常遗漏异步模式、mock 设置
# - 但能发现人类遗漏的边界情况

# 第三步：生成对抗性测试
# "什么输入会破坏这个函数？为这些输入写测试。"

测试维护

AI 擅长：

  • API 变更时更新测试
  • 识别不稳定测试
  • 在不牺牲覆盖率的情况下减少测试套件运行时间
  • 从 API 规范生成集成测试

15.6 AI 调试

调试工作流

  1. 用自然语言描述 Bug — 包括错误信息、预期 vs 实际、最小复现
  2. 提出假设 — 请 AI 按可能性排序列出潜在根因
  3. 请求诊断 — “添加日志/跟踪以缩小问题范围”
  4. 修复并验证 — 应用修复，请 AI 建议本可以捕获该 Bug 的测试用例

有效的调试提示：

"我在这个分布式任务队列中遇到了竞态条件。
这是 worker 代码和错误跟踪。
你会使用哪些同步原语？
请逐步分析导致 Bug 的时间序列。"

"这个 SQL 查询在 500 万行的表上运行需要 12 秒。下面是 EXPLAIN ANALYZE 输出。
哪些索引会有帮助？能否重写查询？"

15.7 AI 时代的安全

AI 生成代码的特定风险

风险频率缓解措施
AI 输出中的提示注入常见永远不要信任处理用户输入的 AI 生成代码
幻觉依赖包罕见但危险验证每个包名、版本、导入路径
不安全默认配置常见显式审查安全配置（CORS、认证、加密）
授权逻辑错误偶尔认证逻辑必须经过人工审查
供应链风险日益增长固定依赖版本，使用 pip audit / npm audit / trivy

LLM 应用的 OWASP Top 10（2025）

  1. 提示注入
  2. 不安全的输出处理
  3. 训练数据投毒
  4. 模型拒绝服务
  5. 供应链漏洞
  6. 敏感信息泄露
  7. 不安全的插件设计
  8. 过度代理权限
  9. 过度依赖（信任 AI 输出的倾向）
  10. 模型盗窃

对软件工程师： #1、#2、#9 是你在每次 AI 交互中的直接责任。

代码审查的安全必须提示

添加到你的审查工作流中：

"分析这段代码，检查：
- SQL 注入（特别是 AI 生成原始查询时）
- XSS 漏洞
- 路径遍历
- 不安全反序列化
- 硬编码密钥或凭据
- 认证/授权绕过
- 泄露内部状态的不当错误处理"

15.8 工程师的 AI 思维模型

何时信任 AI 代码

场景信任级别行动
样板代码（CRUD、API 客户端、配置）高快速审查即可发布
算法实现中充分测试、基准测试
安全关键代码极低全面人工审查，未经测试绝不部署
状态机 / 并发低模型检查、广泛测试
新颖架构无用作灵感，从头编写

“审查密度“法则

你对领域了解越少，就越需要仔细审查 AI 的输出。矛盾的是，初级工程师应比高级工程师更仔细地审查 AI 代码，因为他们缺乏发现错误的模式库。

氛围编程——何时有效，何时无效

有效场景：

  • 原型、MVP、内部工具
  • UI 组件（尤其是有可视化预览时）
  • 数据转换脚本
  • 样板代码和脚手架

危险领域：

  • 生产支付流程
  • 认证和授权系统
  • 医疗或安全关键软件
  • 加密实现
  • 分布式共识

工程师角色的演变

昨天今天明天
写代码引导 AI 写代码定义结果，验证质量
手动调试AI 建议修复，人决定AI 调试和修复，人签字
逐行审查 PR审查 AI 生成的摘要审查 AI 审查过的摘要
设计架构与 AI 共同设计AI 提出架构，人选择

蓬勃发展的工程师不是与 AI 竞争的工程师，而是建立反馈循环的工程师：人的判断 → AI 执行 → 人的验证 → AI 精炼。

趋势很明确： 从如何（命令式编程）到什么（声明式、自然语言）。角色从“写作者“转变为“规格制定者 + 审查者 + 架构师“。你的人本智慧——知道构建什么以及是否正确——变得更加有价值，而非更少。
